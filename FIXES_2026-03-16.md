## Issue #1: MQTT Regex Filter - Backend Implementation ✅

### Problem
The Web UI (`/mqtt/config`) had complete regex filter support, including:
- Filter input field with examples
- Quick-add buttons for common patterns
- Help text and documentation

However, the backend wasn't using the filter value at all.

### Root Cause
1. `Subscription` struct was missing the `filter` field
2. INI parser wasn't loading `FILTER=` lines from config file
3. No filter logic before sending to Miniserver

### Solution
**Files Modified:**
- `crates/mqtt-gateway/src/subscription.rs`

**Changes:**
1. Added `filter: Option<String>` to `Subscription` struct
2. Updated INI parser to load `FILTER=` from config
3. Added `should_filter()` method:
   - Normalizes topic (replace `/` with `_`)
   - Matches against regex pattern
   - Returns `true` if message should be excluded
4. Updated `Relay::send_to_miniserver()` to check filter before sending
5. Added unit tests for filter parsing

**How It Works:**
```rust
// Example subscription with filter
[HomeAutomation]
TOPIC=home/#
NAME=Home Sensors
FILTER=_healthcheck_|_info_|_announce_
ENABLED=1

// When message arrives:
// 1. Topic: "home/sensor/_healthcheck_"
// 2. Normalized: "home_sensor__healthcheck_"
// 3. Regex match: "_healthcheck_" matches
// 4. Result: Message FILTERED (not sent to Miniserver)
```

**Testing:**
```bash
# Add subscription with filter
cat > volumes/config/system/mqtt_subscriptions.cfg << 'EOF'
[Test]
TOPIC=home/#
FILTER=_healthcheck_|_info_
ENABLED=1
EOF

# Publish filtered message (should NOT go to Miniserver)
mosquitto_pub -t "home/_healthcheck_" -m "OK"

# Publish normal message (should go to Miniserver)
mosquitto_pub -t "home/temperature" -m "22.5"

# Check logs for filter activity
docker logs rustylox 2>&1 | grep FILTERED
```

## Issue #3: Plugin Installation File Upload ✅

### Problem
The plugin installation handler was a stub:
```rust
pub async fn install_submit(State(_state): State<AppState>, _multipart: Multipart) -> Html<String> {
    // TODO: Handle file upload and install plugin
    Html(String::from("<div class='success'>Plugin installed successfully...</div>"))
}
```

No actual file handling, no plugin installation.

### Root Cause
Handler was never implemented - probably placeholder from initial UI development.

### Solution
**Files Modified:**
- `crates/web-ui/src/handlers/plugins.rs`

**Changes:**
Implemented complete multipart file upload handler:

1. **Extract uploaded file from multipart form**
   - Iterate through multipart fields
   - Find field named "file"
   - Read bytes from field

2. **Validate file**
   - Check `.zip` extension
   - Verify filename is safe

3. **Save to temporary location**
   - Create `$LBHOMEDIR/tmp/` if needed
   - Write uploaded bytes to temp file

4. **Install plugin**
   - Create `InstallRequest` with temp file path
   - Call `PluginInstaller::install()`
   - Wait for completion

5. **Cleanup**
   - Remove temp file (success or failure)
   - Return success/error HTML

**Code Structure:**
```rust
pub async fn install_submit(
    State(state): State<AppState>,
    mut multipart: Multipart,
) -> Html<String> {
    // Extract file from multipart
    while let Ok(Some(field)) = multipart.next_field().await {
        if field.name() == "file" {
            let filename = field.file_name().unwrap_or("upload.zip");
            let data = field.bytes().await?;
            let temp_path = state.lbhomedir.join("tmp").join(filename);
            tokio::fs::write(&temp_path, &data).await?;

            // Install plugin
            let request = InstallRequest {
                zip_path: temp_path.clone(),
                action: InstallAction::Install,
                force: false,
            };
            let result = plugin_manager.install(request).await;

            // Cleanup temp file
            tokio::fs::remove_file(&temp_path).await;

            return Html(format!("Success: {}", result.name));
        }
    }
    Html("<div class='error'>No file uploaded</div>")
}
```

## Issue #4: Send MQTT Topics to Miniserver ✅

### Problem
The `Relay::send_to_miniserver()` method was a stub:
```rust
pub async fn send_to_miniserver(&self, topic: &str, value: &str) -> Result<()> {
    debug!("Relay to Miniserver: {} = {}", topic, value);
    // TODO: Integrate with MiniserverClient
    Ok(())
}
```

Messages were logged but never actually sent to Miniserver.

### Root Cause
1. Relay didn't have access to Miniserver configuration
2. Relay didn't have MiniserverClient instances
3. No topic-to-virtual-input mapping

### Solution
**Files Modified:**
- `crates/mqtt-gateway/src/relay.rs`
- `crates/mqtt-gateway/src/lib.rs`
- `crates/web-api/src/state.rs`
- `crates/loxberry-daemon/src/main.rs`

**Changes:**

1. **Updated Relay to accept full config**
```rust
pub struct Relay {
    subscription_manager: Arc<SubscriptionManager>,
    config: Arc<RwLock<GeneralConfig>>,  // Added
    clients: RwLock<HashMap<String, Arc<MiniserverClient>>>,  // Added
}
```

2. **Implemented Miniserver client caching**
```rust
async fn get_client(&self, ms_id: &str) -> Result<Arc<MiniserverClient>> {
    // Check cache first
    if let Some(client) = self.clients.read().await.get(ms_id) {
        return Ok(Arc::clone(client));
    }

    // Create new client from config
    let config = self.config.read().await;
    let ms_config = config.miniserver.get(ms_id)?;
    let client = Arc::new(MiniserverClient::new(ms_config.clone())?);

    // Cache for reuse
    self.clients.write().await.insert(ms_id.to_string(), Arc::clone(&client));
    Ok(client)
}
```

3. **Implemented actual Miniserver send**
```rust
pub async fn send_to_miniserver(&self, topic: &str, value: &str) -> Result<()> {
    // Check filter first
    if self.subscription_manager.should_filter(topic) {
        debug!("Message FILTERED: {} = {}", topic, value);
        return Ok(());
    }

    // Get first configured Miniserver
    let ms_id = self.config.read().await
        .miniserver.iter().next()
        .map(|(id, _)| id.clone())
        .ok_or(Error::miniserver("No Miniserver configured"))?;

    // Get or create client
    let client = self.get_client(&ms_id).await?;

    // Map topic to VI parameter (replace / with _)
    let param_name = topic.replace('/', "_");

    // Send to Miniserver
    client.send(vec![(param_name.clone(), value.to_string())]).await?;

    info!("Sent to Miniserver {}: {} = {}", ms_id, param_name, value);
    Ok(())
}
```

4. **Shared config between AppState and MqttGateway**
- Changed `MqttGateway::new()` to accept `Arc<RwLock<GeneralConfig>>`
- Updated daemon to create shared config Arc
- Added `AppState::new_with_shared_config()` method

**How It Works:**
```
1. MQTT message arrives: topic="home/temp", payload="22.5"
2. Transformers process (e.g., boolean conversion)
3. Filter check: "home_temp" doesn't match filter pattern
4. Get Miniserver client (from cache or create new)
5. Map topic: "home/temp" → "home_temp"
6. Send HTTP request: GET /dev/sps/io/home_temp/22.5
7. Miniserver updates virtual input
8. Success logged
```
