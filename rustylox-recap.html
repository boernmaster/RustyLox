04 Decision Log

Switched backup format from tar.gz to ZIP

Why: ZIP is more universally accessible (Windows users, Loxone community). Also simpler code — the zip crate handles streaming better than the tar + flate2 combo for incremental file addition.
Considered: keeping tar.gz for Linux convention, but user base skews Windows + Raspberry Pi.

Renamed all crates from loxberry-* to rustylox-*

Why: Brand differentiation. "LoxBerry" is the original Perl project; this rewrite needs its own identity. Also avoids confusion when both exist on the same system.
Considered: keeping loxberry-* for continuity, but the Docker image was already rustylox so the names were inconsistent.

File-persisted JSON for email/task history (not database)

Why: The task scheduler and email manager are stateless per-request. A database would add connection pool overhead for what amounts to a simple append log. JSON files at data/system/*.json match the existing config pattern.
Considered: SQLite via the database crate, but decided to defer until history volumes justify it.

Hostname-based MQTT client ID instead of PID

Why: Two RustyLox containers on the same broker were kicking each other off — same PID (1) inside Docker means identical client IDs. Using HOSTNAME (container name) guarantees uniqueness.
Considered: UUID-based IDs, but hostname is more debuggable in broker logs.

Auth middleware wraps entire web UI, not per-route

Why: Adding .layer() to every route was error-prone — new routes could accidentally be unprotected. A single middleware layer with a public-path whitelist (/login, /static/, /health) is safer by default.
Considered: per-route guards for granularity, but security-by-default won.

Disabled rate limiting (temporarily)

Why: Rate limiter was causing Docker API failures — couldn't extract real client IPs behind the Docker proxy. TODO in code (web-api/src/lib.rs:204) to re-enable with proper IP extraction.
This is tech debt. Needs proper X-Forwarded-For handling before re-enabling.

06 Mental Model Essentials

  1. Everything flows through AppState. Defined in web-api/src/state.rs, it holds Arc<RwLock<GeneralConfig>>, the MQTT gateway handle, plugin manager, auth service, and all other shared state. Axum extracts it via State(state).
  2. LBHOMEDIR is the root of everything. Defaults to /opt/loxberry in Docker. Config at $LBHOMEDIR/config/system/, data at $LBHOMEDIR/data/system/, logs at $LBHOMEDIR/log/system/. All paths are derived from this single env var.
  3. Config is JSON, subscriptions/transformers are INI. general.json is the master config (serde). MQTT subscriptions use .cfg INI format inherited from original LoxBerry. These two formats co-exist and have different parsing paths.
  4. The web UI and REST API are separate crates but share AppState. web-ui depends on web-api for state types. The daemon wires both routers together. UI routes render Askama templates; API routes return JSON.
  5. MQTT gateway is bidirectional. It bridges MQTT broker ↔ Miniserver via UDP. UDP port 11884 receives from external tools; port 8090 receives from Miniserver Virtual Outputs. Messages flow both directions with transformer pipeline in the middle.
  6. Plugins are identified by MD5 hash. Hash is computed from author + name + folder in plugin.cfg. This is the stable identifier used in URLs, API, and the plugin database. Not the ZIP filename.
  7. Auth middleware wraps the entire app. Public paths (/login, /static/, /health) are whitelisted. Everything else requires a valid lb_token session cookie (set on login, backed by JWT).
  8. cargo fmt before every tag. PRs frequently merge without formatting. CI checks cargo fmt --check. Always run cargo fmt --all on main before creating a release tag, or the Docker build will fail.
  9. Docker builds own the target/ dir. After docker compose build, the target/ directory may be owned by root. Fix with sudo chown -R $USER target/ before local cargo commands.
  10. Crate names use rustylox-*, not loxberry-*. Renamed in v0.6.18. The binary is rustylox-daemon. Prometheus metrics use rustylox_ prefix. Old references in docs may still say "loxberry".

07 Cognitive Debt Hotspots

High

miniserver_backup.rs — 969 lines, heavily changed, minimal tests

The largest file in the project. Implements recursive Miniserver filesystem walking, SSE progress streaming, ZIP archive creation, and backup scheduling. Changed 15+ times in 2 weeks. No dedicated test file.

Suggestion: Extract the filesystem walker and ZIP packer into testable functions. Add at least a unit test for path filtering (the /sys/internal/ skip logic). Consider splitting scheduling into the task-scheduler crate.
High

Low test coverage relative to velocity

34 test functions across 20,509 lines of Rust (~1 test per 600 lines). Only 4 integration test files exist. The auth, email, backup, and task-scheduler crates have zero test files. Given the pace of change (146 commits / 2 weeks), regressions are likely.

Suggestion: Prioritize integration tests for the auth flow (login → cookie → protected route) and the MQTT pipeline (UDP in → transform → MQTT out). These are the highest-traffic code paths.
Medium

Rate limiting disabled with a TODO

The rate limiter in web-api/src/lib.rs:204 is commented out because it couldn't extract client IPs behind Docker's proxy. This leaves the API without brute-force protection despite having auth.

Suggestion: Implement X-Forwarded-For / X-Real-IP extraction in the middleware layer and re-enable the rate limiter. Or use Axum's ConnectInfo with Docker's host network mode.
Medium

web-ui/src/lib.rs changed 22 times — route sprawl

Every new page adds routes to this file. It's becoming a long flat list of .route() calls with no grouping beyond comments. Easy to add a route that bypasses auth middleware if placed wrong.

Suggestion: Group routes into sub-routers by domain (e.g., miniserver_routes(), mqtt_routes()) and nest them. This also makes it clearer where the auth middleware layer applies.
Medium

plugin_web.rs (595 lines) — PHP-CGI bridge complexity

This file handles proxying web requests to legacy PHP plugins via php-cgi. It builds environment variables, manages include paths, handles AJAX requests, and serves static plugin assets. High coupling to LoxBerry SDK conventions with no tests.

Suggestion: Document the expected environment variables and CGI conventions inline. The Vitoconnect plugin is the primary consumer — add a smoke test that exercises the CGI bridge.
Low

Stale branches and Dependabot PRs

2 local branches (claude/add-login-redirect-cookie, feature/docker-entrypoint-permissions) appear to be superseded by work merged to main. 4 Dependabot PRs pending on remote. Minor clutter risk.

Suggestion: Clean up merged branches with git branch -d. Review and merge or close the Dependabot PRs to keep the dependency chain current.
Low

Version number inconsistency in CHANGELOG

Versions jump from 1.3.0 → 0.6.11 → ... → 0.7.0. Early releases used 1.x.x, then the project switched to 0.x.x. This can confuse tooling and users comparing versions.

Suggestion: Consider a note in the changelog explaining the reset, or clean up historical entries to use consistent semver ordering.
